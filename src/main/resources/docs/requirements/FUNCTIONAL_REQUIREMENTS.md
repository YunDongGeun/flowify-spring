# 기능적 요구사항

---

## 1. 개요

### 1.1 시스템 개요

Flowify는 비전문가가 AI 자동화 파이프라인을 코드 없이 구축할 수 있도록 설계된 노드 기반 워크플로우 플랫폼이다.

현재 대부분의 일반 사용자는 ChatGPT 등의 생성형 AI를 일회성 질의응답 도구로만 활용하고 있으며, AI 결과물을 Notion, Google Drive 등 외부 도구로 옮기는 과정에서 반복적인 수작업이 발생하고 있다. 기존 자동화 도구인 n8n, Zapier, Make는 이 문제를 해결할 수 있는 기능을 제공하지만, 노드 개념 이해, 데이터 매핑 설정, OAuth 인증 등 개발자 중심의 설계로 인해 비전문가에게는 높은 진입 장벽이 존재한다.

Flowify는 유사한 목적의 서비스들을 도메인 카테고리 단위로 묶어 노드로 제공한다. 사용자는 일상 언어로 된 질문과 선택지를 통해 원하는 것을 선택하면, 시스템이 내부적으로 적절한 노드 타입(트리거, 조건 분기, Loop 등)을 결정한다. UX 표면에는 기술 용어를 노출하지 않으며, 사용자에게 구체적인 기술 설정을 요구하지 않는다. 데이터 수집 방식, 서비스 간 데이터 규격 통합, LLM 연동 등 기술적으로 복잡한 처리는 시스템 내부에서 자동으로 결정되어 사용자에게 노출되지 않는다. 직접 생성, AI 기반 채팅형 워크플로우 생성, 템플릿 기반 생성을 함께 제공하여 비전문가도 즉시 자동화를 구축할 수 있는 환경을 목표로 한다.

### 1.2 시스템 목표

#### 기능 목표

커뮤니케이션(Gmail, Slack), 저장소(Google Drive, Notion), 스프레드시트(Google Sheets), 웹 수집(쿠팡, 원티드, 네이버 스마트스토어, 네이버 뉴스, 구글 뉴스), 캘린더(Google Calendar) 5개 도메인 노드를 제공한다. 각 노드는 트리거, 필터링, 반복, 조건 분기, AI 처리, 알림 등의 기능을 내부적으로 선택적으로 포함한다. LLM 기반 플로우 자동 생성과 템플릿 기반 워크플로우 생성을 통해 다양한 수준의 사용자가 자동화를 즉시 구축할 수 있도록 한다.

#### 사용성 목표

노드 선택부터 설정 완료까지 대부분의 과정을 마우스 클릭만으로 완료할 수 있도록 설계한다. 모든 노드 설정은 일상 언어 기반의 질문과 선택지 제시 방식으로 진행되며, 사용자에게 기술 용어(트리거, 조건 분기, Loop 등)를 노출하지 않는다. 사용자 직접 입력은 최종 설정 단계에서만 허용하고, 그 외 과정은 선택지 기반으로 구성한다. LLM 응답 대기 중 단계별 실시간 미리보기를 제공하여 사용자 경험을 보완한다.

### 1.3 서비스 철학

#### 대상 사용자

비전문가, 일반인. 자동화나 프로그래밍에 대한 사전 지식이 없는 사용자.

#### 핵심 원칙

1. **시스템 결정 원칙**: 사용자는 "무엇을 원하는지"만 선택한다. 어떤 노드 타입이 되는지는 시스템이 내부적으로 결정한다. 사용자가 서비스 노드를 직접 선택하는 것이 아니라, 사용자의 설정에 따라 노드가 결정된다.
2. **기술 용어 비노출**: UX 표면에 기술 용어(트리거, 조건 분기, Loop 등)를 노출하지 않는다.
3. **일상 언어 인터페이스**: 사용자에게 구체적인 기술 설정을 요구하지 않는다. 일상 언어로 질문하고 선택지를 제공한다.

#### 카테고리 및 서비스 체계

확장성을 열어두고 카테고리 단위로 구성한다. 각 카테고리별로 구체적 서비스를 하나씩 확장해 나간다.

| 카테고리 | 서비스 예시 | 비고 |
|---------|-----------|------|
| 커뮤니케이션 | Gmail, Slack | |
| 저장소 | Google Drive, Notion | |
| 스프레드시트 | Google Sheets | |
| 웹 수집 | 쇼핑몰, 뉴스, 유튜브 | 세분화된 하위 카테고리 |
| 캘린더 | Google Calendar | |
| AI 노드 | LLM 기반 처리 | 중간 과정 전용 |

시작/도착 노드 선택 시 사용자에게는 카테고리를 먼저 보여주고, 카테고리 선택 후 구체적 서비스를 선택하도록 한다. 사용자가 생각하는 데이터 종류나 용도(문서, 이메일, 쇼핑몰 등) 기준으로 자연스럽게 선택할 수 있도록 한다.

#### 데이터 타입 체계

선택지 매핑의 기준이 되는 데이터 타입 분류이다.

| 데이터 타입 ID | 표시 이름 | 설명 |
|-------------|---------|------|
| FILE_LIST | 파일 목록 | 여러 파일이 들어오는 경우 |
| SINGLE_FILE | 단일 파일 | 하나의 파일이 들어오는 경우 |
| TEXT | 텍스트 | AI 요약 결과 등 텍스트 데이터 |
| SPREADSHEET_DATA | 스프레드시트 데이터 | 시트/셀 단위의 구조화된 데이터 |
| EMAIL_LIST | 이메일 목록 | 여러 이메일이 들어오는 경우 |
| SINGLE_EMAIL | 단일 이메일 | 하나의 이메일 데이터 |
| API_RESPONSE | API 응답 데이터 | JSON 형식의 외부 API 응답 |
| SCHEDULE_DATA | 일정 데이터 | 캘린더 일정 정보 |

- **API 응답 데이터 처리**: JSON 응답을 시스템이 미리 매핑하여 필요한 데이터만 추출한다. 추출 대상 예시: 상품명, 평점, 가격, 특정 키워드 등. 웹 수집의 경우 크롤링이 필요할 수 있으므로, 대상 페이지별로 HTML 태그를 미리 정해두어 데이터를 추출한다. 이 크롤링 규칙은 개발팀이 서비스 코드 레벨에서 미리 구현해둔다.
- **데이터 타입 변환**: 노드 처리 결과에 따라 데이터 타입이 변환될 수 있다. 다음 노드 설정 시 변환된 데이터 타입 기준으로 선택지가 제공된다.

#### 선택지 매핑 시스템

각 데이터 타입별로 어떤 선택지를 보여줄지 매핑하는 규칙 체계이다.

**관리 방식: JSON 설정 파일 (`mapping_rules.json`)**

선택지 매핑 규칙은 `docs/mapping_rules.json` 파일에 정의한다. 백엔드 서버가 이 파일을 읽어서 사용자에게 동적 선택지를 제공한다.

**파일 최상위 구조:**
- `_meta`: 버전, 설명, 수정일
- `data_types`: 데이터 타입별 선택지 매핑 (핵심)
- `node_types`: 6가지 노드 타입 정의
- `service_fields`: 서비스별 필드 목록 (데이터 필터용)

**특성:**
- 선택지는 트리 구조가 아니다. 이전 노드의 `output_data_type`에 따라 `mapping_rules.json`에서 해당 데이터 타입의 선택지를 조회한다.
- 각 선택지에 `priority` 값이 있어 낮을수록 상단에 표시된다 (추천 선택지 우선).
- `applicable_when` 조건으로 특정 상황에서만 보이는 선택지를 정의한다 (예: 이미지 파일일 때만 "이미지 내용 설명 생성").
- `follow_up`으로 선택 후 후속 설정 화면을 정의한다. `type: "text_input"`이면 사용자 자유 입력 필드를 제공한다.
- `branch_config`로 조건 분기 노드의 분기 기준 선택을 정의한다 (`multi_select` 지원).
- `options_source`가 `"fields_from_data"` 또는 `"fields_from_service"`이면 런타임에 동적으로 선택지를 생성한다.

**백엔드 처리 흐름:**
1. 사용자가 "+" 버튼 클릭 → 백엔드가 이전 노드의 `output_data_type` 확인
2. `mapping_rules.json`에서 해당 데이터 타입 조회
3. `requires_processing_method`가 true이면 1차 처리 방식(한 건씩/전체) 질문 반환
4. `actions` 배열에서 `applicable_when` 필터링 → `priority` 정렬 → 선택지 반환
5. 사용자 선택 시 `node_type`으로 내부 노드 결정, `follow_up`/`branch_config` 있으면 후속 화면 표시
6. `output_data_type`으로 다음 노드 선택지 준비

**확장 방법:**
- 새 서비스 추가: `service_fields`에 필드 목록 추가
- 새 선택지 추가: 해당 데이터 타입의 `actions`에 항목 추가, `priority`로 순서 조정
- 새 데이터 타입 추가: `data_types`에 새 키 추가

#### 성능 및 안정성 목표

LLM 처리 노드는 비동기 방식으로 실행되어 전체 파이프라인 지연을 방지한다. Loop 노드에는 최대 반복 횟수와 타임아웃을 적용하여 무한 루프를 차단한다. 노드 실행 전후 상태를 스냅샷으로 저장하여 실패 시 롤백 및 재실행이 가능하도록 보장한다.

#### 보안 목표

Google OAuth SSO를 통한 단일 로그인을 지원하며, 서비스별 OAuth 토큰은 암호화하여 user_id 단위로 격리 저장한다. 모든 워크플로우의 소유권은 user_id 단위로 엄격하게 관리한다.

#### 확장성 목표

Strategy 패턴과 Factory 패턴을 적용하여 새로운 노드 추가 시 기존 워크플로우 엔진 코드를 수정하지 않아도 되는 구조를 유지한다. 개방-폐쇄 원칙(OCP)을 준수하여 향후 도메인 노드 및 서비스 연동 확장에 유연하게 대응할 수 있도록 설계한다.

#### UC-W01-D: 직접 설정 상세

| 항목 | 내용 |
|------|------|
| **유스케이스 식별자** | UC-W01-D |
| **유스케이스 명** | 직접 설정 상세 |
| **개요** | 사용자가 워크플로우의 중간 노드를 직접 설정한다. 양측 패널 구조에서 3단계 설정 흐름을 통해 노드를 구성하며, 사용자의 선택에 따라 시스템이 노드 타입을 결정한다. |
| **주 액터** | 사용자 |
| **선행 조건** | - UC-W01-C에서 "직접 설정"이 선택되어야 한다. |
| **후행 조건** | 중간 과정의 모든 노드가 설정되고, 도착 노드의 세부 설정으로 진행할 수 있다. |

**화면 구조:**
- 워크플로우 전체가 노드 연결 형태로 화면에 표시된다.
- 미설정 위치에 "+" 버튼과 "다음 노드를 설정하세요" 텍스트가 표시된다.
- "+" 버튼을 누르거나 기존에 설정된 노드를 선택하면, 양 옆에 설정 창이 표시된다.
  - **왼쪽**: 이전 노드에서 들어오는 데이터 + 설정 선택지
  - **오른쪽**: 현재 노드에서 처리되어 다음으로 넘어가는 데이터
- 완성된 워크플로우에서도 노드 클릭 시 동일한 설정 창이 열려 수정이 가능하다.

**3단계 설정 흐름 패턴:**

| 단계 | 액터 | 시스템 |
|------|------|--------|
| 1 | 사용자가 "+" 버튼을 클릭한다. | |
| 2 | | **1차 — 처리 방식 선택**: 목록형 데이터의 경우 "한 건씩" 또는 "전체 사용"을 먼저 묻는다. "한 건씩" 선택 시 내부적으로 Loop 노드가 된다. |
| 3 | 사용자가 처리 방식을 선택한다. | |
| 4 | | **2차 — 데이터 타입별 선택지**: 분류/분기 선택지, 가공 선택지, 기타 동작 선택지가 함께 표시된다. 추천 선택지는 순서를 위로 올려서 표현한다. |
| 5 | 사용자가 선택한다. | 사용자의 선택이 해당 노드의 타입을 결정한다. |
| 6 | | **3차 — 후속 설정**: 2차에서의 선택에 따라 후속 설정 화면이 결정된다. 기본적으로 선택지를 제공하고, 사용자 직접 입력란은 마지막 설정 단계에서만 제공한다. |
| 7 | 사용자가 후속 설정을 완료한다. | |
| 8 | | 시스템은 노드를 확정하고 다음 "+" 위치를 표시한다. |

**조건 분기 후 흐름:**
- 분기된 각 경로마다 독립적인 "+" 버튼이 표시되고, 각 경로별로 독립적인 설정 흐름이 진행된다.
- 모든 분기 경로가 동일한 도착 노드로 합류할 수 있다.
- 일부 분기 경로만 먼저 설정하고 나머지는 나중에 설정할 수도 있다.

**워크플로우 종료:**
- 사용자가 "+" 대신 도착 노드를 직접 선택하면, 들어오는 데이터에 맞게 도착 노드의 세부 설정 화면이 열린다.
- 설정 완료 시 해당 경로의 워크플로우가 완성된다.

**예외 상황 시나리오:**

| 식별자 | 조건 | 시스템 처리 |
|--------|------|------------|
| E1 | 호환성 불일치 | 중간 과정의 출력 데이터가 도착 노드의 입력과 호환되지 않는 경우 사용자에게 경고를 표시한다. |
| E2 | 이전 단계 변경 | 사용자가 이전 단계로 돌아가서 선택을 변경할 수 있다. 이후에 이미 설정된 노드들은 삭제하지 않고 유지하되, 호환성 경고를 표시한다. |
| E3 | 노드 삭제 | 노드를 삭제하면 해당 노드 이후의 모든 노드가 함께 삭제된다. 이후 노드가 이전 노드의 출력에 의존하므로 유지할 의미가 없기 때문이다. 조건 분기 노드를 삭제할 경우, 모든 분기 경로의 후속 노드가 함께 삭제된다. |
